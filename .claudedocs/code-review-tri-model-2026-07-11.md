### 1. [HIGH] One-shot group reconciliation at init races Chrome's progressive session restore; saved groups are pruned and the destruction is immediately persisted
**File:** `background.js:148` | **Sources:** claude:restart-persistence, grok | **Category:** restart-persistence

Group reconciliation runs exactly once, early in init: background.js:148-149 calls context.state.reconcileWithLiveGroups() with the result of a single chrome.tabGroups.query({}). At browser startup Chrome restores windows/tabs/groups progressively and nondeterministically, so this query can return zero or partial groups while the extension's service worker (which starts immediately) is initializing. reconcileWithLiveGroups then treats every saved group as dead: shared/shadow-state.js:1136-1141 deletes them from this.groups and this.groupColors. The rescue maps (shadow-state.js:1114-1133) are built but there are no live groups yet to rescue onto, so the titles are simply discarded. background.js:170 then calls saveState(), persisting the gutted state to chrome.storage.local — the saved group metadata is now unrecoverable. When Chrome finishes restoring the group seconds later, onGroupCreated fires (Chrome frequently supplies an empty title initially — the codebase's own retryMissingGroupTitles and rescue machinery exist because of this) and addGroup stores an untitled group; retryMissingGroupTitles (background/context.js:133-167) can only poll Chrome, not the already-destroyed saved state. The only safety net, the deferred 2s retry, is gated at background.js:94 on TAB relationship stats: it requires stats.savedRelationships > 0 AND survivingRelationships < 70%. Groups created by right-clicking in the side-panel list and dragging tabs in are typically flat root-level tabs (grouping does not create parent/child lineage), so savedRelationships is 0 and the retry NEVER runs for exactly the user's reported scenario. Even with lineage, high-quality tab matching suppresses the retry while group loss goes completely unmeasured — the gate is blind to groups.

**Failure scenario:** User has 12 root-level tabs, 5 dragged into a titled group 'Research' via the side panel (no parent/child lineage → savedRelationships=0). Chrome restarts; the SW inits while Chrome has restored 3 tabs and 0 groups. tabGroups.query returns [] → all saved groups pruned at shadow-state.js:1136-1141; saveState() at background.js:170 persists the loss. Retry gate at background.js:94 is false (savedRelationships=0) so no second pass. Chrome then restores the group with a new ID and an initially-empty title → the extension shows an untitled group (or, if Chrome's own title restore also lags, the group appears 'gone'). Matches the reported bug exactly.

**Verifier:** CONFIRMED with direct file reads. (1) background.js:148-149: one-shot reconcileWithLiveGroups with a single tabGroups.query in mainline init. (2) shadow-state.js:1136-1141: all saved groups absent from the live set are deleted from this.groups and this.groupColors; the rescue maps (1112-1133) are function-local and the rescue loop iterates liveGroups, so with an empty/partial query result titles are discarded outright. (3) background.js:170 saveState() persists the pruned state to chrome.storage.local (context.js:50-58) — loss is permanent. (4) The only safety net, the 2s retry at background.js:94, is gated on savedRelationships > 0, which counts only parentId lineage (shadow-state.js:803); Chrome tab-group membership (tab.groupId) contributes nothing, so flat tabs dragged into side-panel groups give savedRelationships=0 and the retry never runs — and even when it runs, its gate measures tab-lineage survival, never group survival. (5) No post-prune recovery exists: onGroupCreated (tab-events.js:218-227) just addGroup()s Chrome's payload; retryMissingGroupTitles (context.js:133-167) polls only Chrome, not saved state; checkForCrashRecovery (sessions.js:408-419) is notification-only. The race premise is real MV3 behavior (SW starts at launch, session restore is progressive), and the codebase is its own witness: the 2s tab-reconciliation retry, RETRY_GROUP_TITLE_DELAYS polling, and the push-rescued-titles-back-to-Chrome block at background.js:151-158 all exist because restore is staggered and Chrome-restored groups can lack titles — which also refutes the 'Chrome restores titles itself' defense (that machinery would be dead code). Severity: I rate it high rather than critical — the loss requires the startup timing race to actually occur, and the worst user-visible outcome additionally requires Chrome's own title restore to lag/fail; but when it hits, it is persisted, unrecoverable user-data loss that structurally defeats the extension's dedicated group-persistence feature in exactly the reported flat-grouped-tabs scenario.

### 2. [HIGH] Any error in init() leaves ctx.initComplete=false forever — all tab/group events buffered indefinitely and never drained
**File:** `background.js:194` | **Sources:** claude:restart-persistence, claude:races-storage, grok | **Category:** init-gate

In particular chrome.windows.getCurrent() at background.js:161 rejects with 'No current window' when the SW wakes before any browser window exists (e.g., Chrome auto-started at OS login in background mode, or very early in startup). If any of these throw, initComplete stays false permanently for the lifetime of that SW instance: every handler in background/tab-events.js (lines 31, 61, 91, 149, 179, 193, 202, 219, 229, 254) pushes to pendingEvents and returns, so no tab creation/removal/group event is ever applied or persisted, saveState() at line 170 never runs, and pendingEvents grows unboundedly. The extension is silently dead until Chrome kills and restarts the worker. Since this failure mode fires precisely during browser startup, all the group-restore events Chrome emits during the restore burst are swallowed.

**Failure scenario:** Chrome launches at OS login with the extension SW starting before a window is created. chrome.windows.getCurrent() at background.js:161 rejects → catch at 194 logs and returns → initComplete never set. Chrome then restores 20 tabs and 3 groups; all events are buffered and never drained; nothing is saved. If the SW is later suspended, the buffered events are lost with it, and on the next wake init() reconciles against whatever storage last held — group/tree changes from the entire dead period are gone.

**Verifier:** Confirmed with direct file evidence. background.js:190-191 (ctx.initComplete = true; tabEvents.drainPendingEvents()) are the final statements inside init()'s try, after 8+ awaitable calls including chrome.windows.getCurrent() at :161; the catch at :194-196 only logs, with no finally/retry/fallback. Every handler in background/tab-events.js (:31,:61,:91,:149,:179,:193,:202,:219,:229,:254) buffers into an uncapped pendingEvents array when !initComplete, and message-handlers.js:74 gates side-panel messages on the same flag (initDone resolves even on failure, so awaiting it doesn't help). saveState() at :170 is also inside the try, so nothing persists for the lifetime of a failed-init SW instance. Real MV3 semantics support the trigger: chrome.windows.getCurrent() rejects with 'No current window' from a service worker when zero browser windows exist (Chrome background-mode auto-start at OS login; macOS all-windows-closed wake via the extension's own auto-save alarm registered at :173/:240; early-startup race), and tabs.query({currentWindow:true}) at :164 plus the storage reads share the same fragile catch. Self-healing is weak: buffered events reset the SW idle timer, so an active session keeps the dead instance alive indefinitely, and post-restart reconciliation cannot recover openerTabId lineage or group changes from the dead period. Severity: high — a single rejection silently disables the entire event pipeline and persistence for the SW instance lifetime, firing precisely during the startup restore burst in the trigger scenario; only mitigation is that the common happy-path startup (window already present) succeeds, which limits frequency but not impact.

### 9. [HIGH] [Codex] Pass 1 tab reconciliation trusts a literal tabId match as authoritative — Chrome tab IDs restart from a low counter each browser launch, so cross-session ID collisions are plausible, not theoretical
**File:** `shared/shadow-state.js:837` | **Sources:** codex | **Category:** schema-drift

reconcileWithLiveTabs Pass 1 (shared/shadow-state.js:837-843) treats `liveById.has(id)` (i.e., a live tab exists with the exact same numeric id as a saved tab) as a confirmed same-session match, skipping any URL/title/window validation entirely. Chrome assigns tab IDs from a counter that restarts near the beginning of each new browser process — meaning a saved tab's old id (e.g., 5) can coincide with an entirely unrelated tab's new id in the freshly-launched session. Unlike Pass 2/2b/3, which all disambiguate via URL, title, or position plus the pickReconcileCandidate refuse-on-ambiguity logic, Pass 1 has zero validation beyond id equality.

**Failure scenario:** A saved tab with id=7 was part of a group and had 2 children (a small subtree). After a full Chrome restart, the new session happens to assign id=7 to a completely unrelated newly-opened tab (plausible since both sessions' id counters start low). Pass 1 matches them purely on id, so `matchedLiveIds.add(7)`. In the later "(b) Update existing / add new" pass, this node's url/title/favIconUrl/groupId/windowId are overwritten with the unrelated live tab's data — while its parentId, children array, and tree position (established by the OLD, unrelated saved tab) are preserved. The result: the group/tree the old tab-id-7 belonged to now silently contains a completely different tab's content, while the actual restored tab that logically continues the old tab-id-7's identity (if any) is treated as an entirely separate, unmatched tab. This can manifest as a group "losing" its expected member (replaced by unrelated content) or as duplicate/orphaned entries elsewhere in the tree — directly consistent with the reported "groups are simply gone" symptom for groups whose member tab ids happen to collide.

**Verifier:** CONFIRMED with file:line evidence; refutation attempts failed on every axis.

1) The code does exactly what the finding says. Pass 1 (shared/shadow-state.js:838-843) is `for (const id of this.tabs.keys()) { if (liveById.has(id)) { matchedLiveIds.add(id); ... } }` — bare numeric ID equality, no URL/title/window validation, no pickReconcileCandidate. Pass 2/2b/3 (lines 868-971) all validate content or position and refuse ambiguity; Pass 1 alone does not.

2) The corruption path is real. A Pass-1-collided saved node is excluded from unmatchedSaved (line 848 checks `!liveById.has(id)`), so URL-based rescue never runs for it. In the update pass (lines 1026-1045), `this.tabs.has(tab.id)` → `updateTab(tab.id, changes)` overwrites url/title/favIconUrl/groupId/windowId with the unrelated live tab's data while preserving the node's parentId/children lineage. The genuinely restored counterpart tab (new ID) falls to `addTab(...parentId: null...)` at line 1043 as a fresh root. This matches the finding's failure scenario precisely.
